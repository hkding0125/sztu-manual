# 参与贡献

这本手册能不能长期有用，取决于后面有没有人继续补。信息一旦过时，手册就会很快变成另一种宣传册。所以如果你发现写错了、过期了，或者你自己刚好踩过某个坑，欢迎把东西补上来。

## 怎么提交

Fork [本项目仓库](https://github.com/hkding0125/sztu-manual)，在 `docs/guide/` 目录下修改或新增 Markdown 文件，然后提交 Pull Request，简单说明你改了什么、为什么改。审核后会合并到主分支。

## 关于写作

最有用的稿子，一般都不是“我总结出三点经验”，而是把你当时到底遇到了什么、怎么选的、后面付了什么代价写清楚。

写的时候可以先记住这几条：

- 先写处境，再写判断。别一上来“先说结论”。
- 多给具体细节。时间、节点、动作，比大词有用。
- 可以有立场，但别假装替所有人发言。
- 少一点“这让我明白了一个道理”，多一点“我当时为什么会这样选”。

后来的人来这里，不是为了听标准答案。他们更想知道：你是谁，你当时卡在哪，你为什么那么选，那套经验放到我身上还管不管用。

## 一个够用的访谈 / 经验稿模板

如果你不知道怎么起笔，可以直接照这个顺序写：

1. **你是谁**
   - 专业 / 年级 / 最后去了哪
   - 如果你愿意，也可以写大概背景，比如绩点区间、科研经历、实习经历
2. **你当时卡在哪**
   - 比如：留学还是读研，继续做材料还是转代码，先进组还是先保成绩
3. **你最后怎么选的**
   - 写真实原因，不用写成完美答案
4. **后来发生了什么**
   - 收获是什么，代价是什么，哪些坑是你当时没想到的
5. **如果重来一次**
   - 你会更早做什么，不会再拖什么
6. **一句留给后来人的话**
   - 不用正确，只要别假

## 一个简单的判断标准

如果一篇稿子读完之后，只能让人觉得“你很厉害”，那它大概率还不够有用。

更好的状态是：后来的人能读出来你的处境、你的代价、你的判断边界，然后自己决定要不要参考你。这种稿子通常不那么漂亮，但会更真。
